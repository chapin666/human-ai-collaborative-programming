# 改写示例3：实践工具部分

## 原文（问题版本）

```markdown
### 实践工具

值得注意的是，写好提示词对于获得高质量的AI输出非常重要。让我们来看一下一些实用的提示技巧。

首先，我们需要给AI一个明确的角色。这样可以让AI更好地理解我们的需求。其次，我们要提供足够的上下文。这是非常重要的。最后，我们要明确约束条件。

这是一个很好的实践。希望大家能够掌握这些技巧，提升自己的AI使用效率。
```

## 问题分析

| 问题 | 位置 | 说明 |
|:---|:---|:---|
| "值得注意的是" | 第1句 | AI套话 |
| "让我们来看一下" | 第2句 | 引导语 |
| "非常重要" | 第4、6句 | 空洞强调 |
| "这是一个很好的实践" | 第7句 | 空洞赞美 |
| "希望大家能够" | 第7句 | 弱化语气 |
| 无具体模板 | 全文 | 抽象建议，无法执行 |
| 无示例 | 全文 | 只有原则，没有演示 |

## 改写版本

```markdown
### 实践工具

**RICE框架：提示工程的四要素**

把模糊需求变成AI可执行的指令，需要四个要素。缺一不可。

#### 1. Role（角色）

**原则**：让AI知道它应该是谁。

**模板**：
```
你是一位[资深后端工程师/架构师/安全专家]，擅长[技术领域]。
你的任务是为[目标读者]提供[输出类型]。
```

**对比**：
- ❌ "帮我看看这段代码"
- ✅ "你是一位有10年经验的Go工程师，擅长高并发系统设计。请审查以下代码的性能和并发安全性"

#### 2. Input（输入）

**原则**：提供决策所需的全部上下文。

**检查清单**：
- [ ] 代码/文档/错误信息（完整粘贴）
- [ ] 当前环境（语言版本、框架版本）
- [ ] 业务背景（这个功能是做什么的）
- [ ] 已尝试的方案（避免重复建议）

**示例**：
```
【代码】
func GetUser(id int) (*User, error) {
    // 具体代码...
}

【环境】
- Go 1.21
- PostgreSQL 15
- 并发量：1000 QPS

【业务】
用户查询接口，要求响应时间 < 50ms

【已尝试】
加了索引，但高并发时仍有延迟
```

#### 3. Constraint（约束）

**原则**：明确不能做什么，比能做什么更重要。

**常用约束模板**：
| 约束类型 | 示例 |
|:---|:---|
| 技术约束 | "不要使用第三方库" |
| 性能约束 | "时间复杂度控制在O(n)" |
| 风格约束 | "遵循Go官方代码规范" |
| 输出约束 | "返回JSON格式，包含error字段" |

#### 4. Expectation（期望）

**原则**：定义"好"的标准。

**模板**：
```
请提供：
1. [具体输出1]
2. [具体输出2]
3. [可选] 对于[关键决策点]的权衡分析

质量要求：
- [可量化的标准1]
- [可量化的标准2]
```

**完整示例**：
```
你是一位资深Go工程师，擅长数据库性能优化。

【代码】
...（完整代码）...

【环境】Go 1.21 + PostgreSQL 15，1000 QPS
【业务】用户查询接口，要求 < 50ms
【约束】不改变表结构，不使用缓存

请提供：
1. 性能瓶颈分析
2. 优化后的代码
3. 优化前后的benchmark对比预测
```

**常见错误**：
- ❌ 一次性提5个需求，AI只能做好1个
- ❌ 没有约束，AI给出不切实际的方案
- ❌ 期望模糊，"写得好一点"无法执行
